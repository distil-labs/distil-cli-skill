# Test-Set Approval

Hard gate in the traces workflow. Run after every trace processing pass (`upload-traces` and `reprocess-traces`), before teacher evaluation. Do **not** run in the dataset workflow — there the user provided the test set and owns it. Also skip in the traces workflow when the user supplied their own `test.jsonl` alongside the traces (same reason — the test set wasn't generated by relabeling).

This step subsumes what the traces workflow previously called "Analyze Original Model Performance". The user cannot meaningfully approve a generated test set without seeing how the production model that produced the traces performs on it — so the original-model scores and analysis are part of the approval evidence, not a separate downstream step.

**In outline:** download the uploads object and original-model predictions → present test-set composition alongside the Original Model Analysis Report and sampled correct/incorrect example tables → block until the user replies 'approved'. The sections below detail each step.

## Inputs

- **Working directory:** the current `iteration-N/` dir (owned by `workflows/improving-a-model.md`'s Iteration Discipline section). On the first upload, this is `iteration-1/`.
- **Test-set composition:** `distil model download-data <model-id>` — pulls the processed `train`, `test`, and `unstructured` splits locally. See `references/tasks/upload-dataset.md`.
- **Original-model predictions on the test set:** `distil model download-traces-predictions <model-id> --file-name original-model-predictions.jsonl`. See `references/tasks/retrieve-predictions.md`.

## What to Present

Present the two blocks below in one message, then ask for approval. Keep it scannable — tables and short prose, not walls of text.

### (1) Generated test set

- **Total size:** <N> examples
- **Label / answer distribution:**
  - Classification → count per class
  - QA → answer-length distribution (min / median / max chars)
  - Tool calling → distribution across tool names
- **Field-length percentiles (p10 / p50 / p90 / max):** for `question`, `context` (if present), `answer`
- **Edge-case coverage:** does every class / tool / scenario named in `job_description.json` appear at least once? List any that do not.
- **Provenance:** how many examples came from relabeled traces vs. any user-provided `test.jsonl`.

### (2) Original-model scores and analysis

Render the **Original Model Analysis Report** template from `references/tasks/analyze-predictions.md` inline — do not duplicate the template here. Fill it in using the downloaded predictions against the relabeled ground truth. Save the filled report to `<iteration-N>/original-model-analysis.md`.

In addition, include two sampled tables. These are the load-bearing part of the approval decision — they give the user concrete signal about the test set quality.

**5–10 examples where the original model is CORRECT** (LLM-as-a-Judge passed, or exact match, depending on task):

| # | Question (truncated) | Original Output | Relabeled Answer |
|---|----------------------|-----------------|-------------------|
| 1 | ... | ... | ... |

**5–10 examples where the original model is INCORRECT** (gives a sense of failure modes and where relabeling moved the goalposts):

| # | Question (truncated) | Original Output | Relabeled Answer | Cause |
|---|----------------------|-----------------|-------------------|-------|
| 1 | ... | ... | ... | <disagreement type, e.g., "original picks 'general', relabeled more specific"> |

Sample the tables across the distribution — don't just take the first N rows.

## Decision Gate

After presenting both blocks, ask the user verbatim:

```
Reply 'approved' to proceed to teacher evaluation, or tell me what to change (e.g., reprocess with different trace_processing parameters, drop specific examples, add a curated test.jsonl).
```

Do not run `distil model run-teacher-evaluation` until the user replies with approval. If the user asks for changes, the common levers are:

- Adjust `trace_processing` params and run `distil model reprocess-traces`
- Provide a curated `test.jsonl` and re-run `distil model upload-traces`
- Fix upstream issues in `job_description.json` and re-run `distil model upload-traces`

Any of these re-triggers this approval step (and, optionally, the deep-dive Analyze Uploads step).

## Log

Append a log entry at `model-building-log-<name>.md`:

- Verdict: n/a (informational gate)
- Headline: original-model primary-metric score on this test set
- Why: one sentence on what the user agreed to, or what they asked to change

See `references/tasks/maintain-run-log.md`.
